# 🔍 Audit Report: 401 Unauthorized on `/api/assistant/chat`

**Date:** 2025-01-21  
**Issue:** `/api/assistant/chat` returns 401 Unauthorized while `/api/auth/me` works (200 OK)  
**Status:** 🔴 **CRITICAL BUG IDENTIFIED**

---

## Executive Summary

The `/api/assistant/chat` endpoint uses `requireUserOrGuest` middleware which has a **critical bug**: it only handles guest tokens from the Authorization header, but **fails to handle regular user tokens** sent via Authorization header. This causes authenticated users to receive 401 errors even though their tokens are valid.

---

## Root Cause Analysis

### 1. **Token Structure Differences**

**Regular User Tokens** (used by `/api/auth/me`):
- Signed with: `{ userId: string }`
- No `role` field
- Generated by: `generateToken(userId)` in `middleware/auth.js`
- Example: `jwt.sign({ userId: "cmj4avaku0000jvbohg39rsvw" }, JWT_SECRET)`

**Guest Tokens** (for unauthenticated users):
- Signed with: `{ guestId: string, role: 'guest' }`
- Has `role: 'guest'` field
- Generated by: `assistant.js` route `/api/assistant/guest`
- Example: `jwt.sign({ guestId: "guest_xxx", role: 'guest' }, JWT_SECRET)`

### 2. **Middleware Comparison**

#### `/api/auth/me` uses `requireAuth` middleware:
**File:** `apps/core/cardbey-core/src/middleware/auth.js` (lines 39-132)

```javascript
export async function requireAuth(req, res, next) {
  const token = extractToken(req); // Gets token from Authorization header
  const decoded = jwt.verify(token, JWT_SECRET);
  const user = await prisma.user.findUnique({ where: { id: decoded.userId } });
  // ✅ Works correctly for user tokens
}
```

**Behavior:**
- ✅ Extracts token from Authorization header
- ✅ Verifies JWT
- ✅ Looks up user by `decoded.userId`
- ✅ Works correctly

#### `/api/assistant/chat` uses `requireUserOrGuest` middleware:
**File:** `apps/core/cardbey-core/src/middleware/guestAuth.js` (lines 19-97)

```javascript
export async function requireUserOrGuest(req, res, next) {
  // Step 1: Try cookie token (full user)
  const cookieToken = req.cookies?.accessToken;
  if (cookieToken) { /* ... verify user ... */ }
  
  // Step 2: Try Authorization header
  const authHeader = req.headers.authorization;
  if (authHeader && authHeader.startsWith('Bearer ')) {
    const token = authHeader.substring(7);
    const decoded = jwt.verify(token, JWT_SECRET);
    
    if (decoded.role === 'guest') {
      // ✅ Handles guest tokens correctly
      req.isGuest = true;
      return next();
    }
    // ❌ BUG: If token is a regular user token (has userId, no role),
    //    it doesn't match the guest check and falls through
  }
  
  // ❌ Returns 401 even though token is valid
  return res.status(401).json({ error: 'Authentication required' });
}
```

**Behavior:**
- ✅ Checks cookie token first (but frontend doesn't use cookies)
- ✅ Checks Authorization header
- ✅ Handles guest tokens (`decoded.role === 'guest'`)
- ❌ **BUG: Does NOT handle regular user tokens from Authorization header**
- ❌ Falls through to 401 even with valid user token

### 3. **The Bug Flow**

1. Frontend sends: `Authorization: Bearer <user-token>` (valid JWT with `{ userId }`)
2. `requireUserOrGuest` extracts token from Authorization header ✅
3. JWT verification succeeds ✅
4. Checks `if (decoded.role === 'guest')` → **FALSE** (user token has no `role` field)
5. Code falls through without handling user token ❌
6. Returns 401 Unauthorized ❌

## Additional Findings

### 1. Cookie Token Check Issue
The middleware checks `req.cookies?.accessToken` but the frontend doesn't use cookies for auth. This is a dead code path that won't execute.

### 2. Inconsistent Token Sources
- `requireAuth` uses `extractToken()` which checks Authorization header, query param, and cookies
- `requireUserOrGuest` only checks cookies first, then Authorization header
- This inconsistency could cause issues

### 3. Error Handling
The middleware catches JWT verification errors but doesn't distinguish between:
- Invalid token format
- Expired token
- Wrong secret
- Missing userId/guestId

Better error messages would help debugging.

---

## Conclusion

The bug is a **logic error in `requireUserOrGuest` middleware**: it only handles guest tokens from the Authorization header but fails to handle regular user tokens. The fix is straightforward - add a check for `decoded.userId` after the guest token check.
