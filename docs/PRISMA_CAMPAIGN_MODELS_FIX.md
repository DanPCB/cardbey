# Prisma campaign models mismatch – root cause and fix

## Root cause

The core server uses a **single** Prisma client from `node_modules/.prisma/client-gen`, which is generated by `prisma generate`. Both SQLite and Postgres schemas use the **same** output path:

- `prisma/sqlite/schema.prisma` → `output = "../../node_modules/.prisma/client-gen"`
- `prisma/postgres/schema.prisma` → `output = "../../node_modules/.prisma/client-gen"`

So **whichever schema was used last for `prisma generate`** is what the server gets. Campaign routes expect these delegates on the client: `campaignPlan`, `campaignValidationResult`, `campaignV2`, `campaignScheduleItem`, `creativeCopy`, `creativeAsset`, `offer`, `channelDeployment`.

**What was going wrong:**

1. **Stale client:** `prisma generate` was not run after campaign models were added to the schema, so the existing client-gen had no campaign delegates. Then `prisma.campaignPlan` (etc.) is `undefined` and `prisma.campaignPlan.findFirst` throws "Cannot read properties of undefined (reading 'findFirst')".
2. **Wrong schema:** If someone ran `prisma generate` from a different project or an old schema path that doesn’t include campaign models, client-gen would not have those delegates.
3. **DATABASE_URL vs client:** The **database** (SQLite or Postgres) is chosen by `DATABASE_URL`. The **client** (which models exist) is chosen by which schema you ran `prisma generate` for. They must match: SQLite URL → generate from sqlite schema (and use `db push`); Postgres URL → generate from postgres schema (and use `migrate deploy`).

## Commands to run locally

From repo root or from `apps/core/cardbey-core`:

**If you use SQLite (default local dev, `DATABASE_URL=file:...` or `.db`):**

```bash
cd apps/core/cardbey-core
npx prisma generate --schema prisma/sqlite/schema.prisma
npx prisma db push --schema prisma/sqlite/schema.prisma
# Restart the core server
```

Or use package scripts:

```bash
cd apps/core/cardbey-core
pnpm run db:generate:sqlite
pnpm run db:push
# Restart the core server
```

**If you use Postgres (`DATABASE_URL=postgresql://...`):**

```bash
cd apps/core/cardbey-core
npx prisma generate --schema prisma/postgres/schema.prisma
npx prisma migrate deploy --schema prisma/postgres/schema.prisma
# Restart the core server
```

Or:

```bash
cd apps/core/cardbey-core
pnpm run db:generate:postgres
pnpm run prisma:migrate:postgres
# Restart the core server
```

After that, the client-gen will include campaign models and the DB will have the campaign tables. Restart the core server so it loads the new client.

## Manual QA checklist

- [ ] **Startup:** Start core server. If campaign models were missing, you see a clear error and the process exits. After running the commands above and restarting, server starts and (in dev) you see `[DB] Prisma client-gen campaign models (dev): { campaignPlan: 'ok', campaignV2: 'ok', campaignScheduleItem: 'ok' }`.
- [ ] **GET /api/campaign/plan?missionId=...**  
  - After running a mission that completed validate-scope, call with valid `missionId`.  
  - Expect **200** and a plan in the response (or 200 with empty plan if none yet). No 500.
- [ ] **POST /api/campaign/create-from-plan**  
  - With valid `planId` and auth.  
  - Expect **200** with `campaignId` in the body. No 500 and no "Cannot read properties of undefined".
- [ ] **GET /api/campaign/by-mission?missionId=...**  
  - After create-from-plan for that mission.  
  - Expect **200** with campaign data (or 404 if no campaign). No 500.

## Which schema is active at runtime?

- **Client (models in code):** Determined by the last `prisma generate` you ran. Check the comment at the top of `node_modules/.prisma/client-gen/schema.prisma` (e.g. "SQLite" vs "Postgres") if you need to confirm.
- **Database (tables):** Determined by `DATABASE_URL` in `.env`. `file:` or `.db` → SQLite; `postgresql://` or `postgres://` → Postgres. Use SQLite schema + `db push` for SQLite; use Postgres schema + `migrate deploy` for Postgres.
