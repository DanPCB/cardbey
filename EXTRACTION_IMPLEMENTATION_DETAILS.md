# Menu Extraction Implementation Details

## Backend Extraction Method

### Primary Engine: **OpenAI Vision API (GPT-4o)**

**Location**: `apps/core/cardbey-core/src/ai/engines/openaiVisionEngine.js`

**Model**: `gpt-4o` (Vision-capable model)

**Flow**:
1. `runOcr(imageUrl)` → `apps/core/cardbey-core/src/modules/vision/runOcr.js`
2. → `openaiVisionEngine.analyzeImage({ imageUrl, task: 'menu' })`
3. → OpenAI Chat Completions API with vision capability
4. → Returns raw text response (or refusal message)

### Current Prompt for Menu OCR

**File**: `apps/core/cardbey-core/src/ai/engines/openaiVisionEngine.js` (line 181)

```javascript
prompt = 'Extract all visible text from this menu image. This is a menu with food items, names, and descriptions. Return ONLY the text content you can see, line by line, exactly as it appears. Do not refuse or say you cannot read it - return whatever text is visible, even if partial.';
```

**Note**: Despite the prompt explicitly saying "Do not refuse", GPT-4o still sometimes returns refusal messages.

### Where Refusal Messages Come From

**Source**: **OpenAI Vision API (GPT-4o) itself**

The refusal messages like:
- "I'm unable to extract text from this image"
- "The image does not contain any visible text"
- "The image is too faint to extract any visible text"
- "I can't assist with that"

These are **direct responses from GPT-4o**, not generated by our code. The model sometimes refuses to extract text even when:
1. The prompt explicitly asks it not to refuse
2. There is visible text in the image
3. The image is clear and readable

### Current Refusal Detection

**Location**: `apps/core/cardbey-core/src/services/menuOcrSingleItem.js` (function `parseOcrText`)

**Strategies**:
1. **Refusal markers list**: Checks for common refusal phrases
2. **Starts with refusal**: If text starts with "I'm unable", "I can't", etc.
3. **Multiple markers**: If text contains 2+ refusal markers
4. **Long refusal patterns**: Specific combinations of refusal phrases
5. **Short + marker**: If text < 100 chars and contains any refusal marker
6. **"Feel free to ask"**: Pattern matching for polite refusals

### Alternative Extraction Methods

**Structured Extraction** (used as fallback/primary for bulk):
- **Location**: `apps/core/cardbey-core/src/engines/menu/extractMenuStructured.js`
- **Method**: Also uses OpenAI Vision API but with structured JSON output
- **Prompt**: Asks for JSON format with items array
- **Advantage**: Less likely to refuse (asks for structured data, not text transcription)
- **Disadvantage**: Sometimes only finds 1 item when there are multiple

**SAM3-Guided OCR** (experimental):
- **Location**: `apps/core/cardbey-core/src/lib/sam3TextRegion.js`
- **Method**: Uses SAM-3 to detect text regions, then runs OCR on those regions
- **Status**: Currently implemented but may not be fully working

### Image Handling

**Private URLs** (192.168.x.x, localhost, etc.):
- Automatically detected by `isPrivateUrl()`
- Fetched and converted to base64 data URL
- Sent to OpenAI as `data:image/jpeg;base64,...`

**Public URLs**:
- Sent directly to OpenAI Vision API

### API Configuration

**Environment Variables**:
- `OPENAI_API_KEY`: Required for OCR to work
- `DEBUG_VISION`: Set to `'true'` for detailed logging

**Model Settings**:
- Model: `gpt-4o`
- Max tokens: `2000`
- Timeout: `30000ms` (30 seconds)
- Max retries: `2`

### Current Issues

1. **Refusal messages still getting through**: Despite prompt and detection, some refusals are accepted as valid names
2. **Grid detection over-segmenting**: Single-item images detected as 9 regions
3. **SAM-3 cutout failing**: Returns 0 regions even for clear product photos
4. **Structured extraction finding only 1 item**: When menu has multiple items, structured extraction sometimes only finds one

## Summary for Implementation Blueprint

**Extraction Engine**: OpenAI Vision API (GPT-4o)
**Refusal Source**: GPT-4o model responses (not our code)
**Current Prompt**: Explicitly asks model not to refuse, but refusals still occur
**Detection**: Multi-strategy pattern matching in `parseOcrText()`
**Issue**: Some refusals still slip through as valid names

**Recommendation**: 
1. Improve prompt to be more directive
2. Add confidence scoring to OCR responses
3. Implement retry logic with different prompts
4. Add UI states to show OCR confidence/refusal status
5. Consider using structured extraction as primary method (less refusals)
